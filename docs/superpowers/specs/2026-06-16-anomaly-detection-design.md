# Anomaly / Info-Attack Detection — design

> Расширение слоя сюжетов Echo-v1 (см. [[story-timeline]] спеку
> `2026-06-15-story-timeline-design.md`, §8). Помечает сюжет как аномальный, когда
> в последнем временном бакете совпадают всплеск объёма и резкий негатив/приток
> источников. Дата: 2026-06-16. Ветка: `feat/story-timeline` (продолжение).

## 1. Цель и границы

Фундамент сюжетов (`incident → story → story_points`) и поле `stories.is_anomaly`
уже есть. Эта фича **заполняет `is_anomaly`** правилом поверх почасовых точек
таймлайна и **подсвечивает** аномальные сюжеты в дашборде. Гражданский кейс —
репутационные кризисы и скоординированные вбросы против бренда.

**Решения (брейншторм 2026-06-16):**
- Правило, не ML; считаем по `story_points` (без LLM).
- **Триггер** = всплеск объёма (обязателен) **И** (резкий сдвиг тональности **ИЛИ**
  приток источников).
- **Подача**: бейдж ⚠ на элементах списка «Сюжеты» + сортировка аномальных
  наверх. На детали сюжета индикатор «⚠ аномалия» уже есть.

**Вне объёма:** LLM-отчёт по алерту, отдельный журнал алертов, пуш-уведомления.

## 2. Модуль `backend/radar/anomalies.py`

Отдельный модуль (одна ответственность, не раздуваем `stories.py`).

```
detect_anomaly(session, story_id) -> bool
```
- Грузит `story_points` сюжета, упорядоченные по `bucket_start`.
- Последний бакет = «текущее»; предыдущие = базовая линия.
- Выставляет `story.is_anomaly` (True/False) и возвращает результат.
- Идемпотентно: при каждом пересчёте переоценивается; флаг может и сняться.

### Правило (пороги — env, подбор эмпирически)
- **Минимум истории.** Если предыдущих бакетов < `ANOMALY_MIN_BUCKETS` (умолч. 3) —
  возвращаем False (базовую линию не построить). Решает «нужно накопить данные».
- **Базовая линия** = среднее по предыдущим бакетам отдельно для `mention_count`,
  `avg_sentiment` (игнорируя None), `source_count`.
- **Всплеск объёма (обязателен):**
  `last.mention_count >= baseline_volume * ANOMALY_VOLUME_FACTOR` (умолч. 3.0)
  И `last.mention_count >= ANOMALY_MIN_VOLUME` (умолч. 3) — чтобы 0→1 не считалось.
- **Сдвиг тональности:**
  `baseline_sentiment - last.avg_sentiment >= ANOMALY_SENT_DROP` (умолч. 0.4) —
  резкий уход в негатив. Если `last.avg_sentiment` is None — признак не выполнен.
- **Приток источников:**
  `last.source_count >= baseline_sources * ANOMALY_SOURCE_FACTOR` (умолч. 2.0).
- **Триггер:** `всплеск AND (сдвиг_тональности OR приток_источников)`.

Базовые средние, равные 0, не должны давать ложный триггер: при `baseline_volume == 0`
порог объёма опирается на абсолютный `ANOMALY_MIN_VOLUME`; при `baseline_sources == 0`
приток источников считается невыполненным (нет базы для кратного роста).

## 3. Интеграция

В `stories.update_stories`, в цикле пересчёта по задетым сюжетам, **после**
`_recompute_points(sid)` вызвать `anomalies.detect_anomaly(session, sid)`. Story-слой
уже обёрнут best-effort в `scheduler._run_brand_pipeline` (падение не ломает
classify/draft). Импорт `anomalies` внутри `stories.py`.

## 4. API

`GET /stories`: сменить сортировку на `Story.is_anomaly.desc(), Story.last_seen_at.desc()`
— аномальные наверх. Поле `is_anomaly` уже в `StoryOut`. Прочие эндпоинты без изменений.

## 5. Фронтенд (echo-app)

В `components/app/Stories.jsx`, в элементе списка `StoriesScreen`, при `s.is_anomaly`
показать красный ⚠-маркер рядом с заголовком. Сортировка приходит с бэкенда. CSS —
класс в `stories.module.css`. Деталь (`StoryDetail`) уже рендерит «⚠ аномалия».

## 6. Тесты

Backend (`test_anomalies.py` + дополнение к `test_stories_api.py`):
- срабатывает при всплеске объёма + резком негативе (≥ min history);
- срабатывает при всплеске + притоке источников (без негатива);
- НЕ срабатывает при всплеске без второго признака;
- НЕ срабатывает при недостатке истории (< `ANOMALY_MIN_BUCKETS`);
- снимается (False) при возврате последнего бакета к базовой линии;
- интеграция: `update_stories` проставляет `is_anomaly` на аномальном сюжете;
- API: `GET /stories` отдаёт аномальные первыми.

## 7. Риски

- **Пороги — эмпирика.** Дефолты — стартовые; калибровать на реальных брендах
  (см. [[echo-testing-real-brands]]). На малом трафике аномалии редки — это ожидаемо.
- **Только последний бакет** как «текущее» — простой MVP-подход; скользящее окно из
  нескольких последних бакетов можно добавить позже без смены интерфейса.
- **Направление сдвига** — фиксируем «уход в негатив» (репутационный кейс); рост
  позитива аномалией не считаем.
