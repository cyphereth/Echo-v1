# Умные и чинимые реплаи — дизайн

Дата: 2026-06-27
Ветка: `feat/intel-thread-context`

## Проблема

Две наблюдаемые боли с реплаями оказались одним корнем:

1. **Дубли при realtime.** Одно и то же сообщение чата сохраняется дважды.
2. **Цепочка «в ответ на» рвётся / неполная.** Родитель не находится, `thread_root_id`
   неверный, контекст ветки пустой.

### Корневая причина A — рассинхрон namespace у групп без `@username`

`post_id` чата — составной: `<namespace>/<msgid>`. Для приватной супергруппы
(добавленной по invite-ссылке, резолвящейся в `#<id>`) два пути считают namespace
по-разному:

| Путь | namespace | пример |
|------|-----------|--------|
| Поллер (`search_chat` → `_parse_tg_chat_message`) | `int(handle[1:])` — **unmarked** peer_id из `#<id>` (`join_invite` отдаёт `chat.id` без префикса) | `1234567890/567` |
| Realtime (`_on_message`) | `str(event.chat_id)` — **marked** id Telethon (префикс `-100`) | `-1001234567890/567` |

Одно сообщение → два разных `post_id` → UNIQUE-ключ `(platform, post_id)` не срабатывает
→ дубль. Цепочка рвётся по той же причине: `_parent_post_id` ищет родителя в namespace
самого упоминания, поэтому realtime-реплай (`-100…/X`) ищет родителя `-100…/Y`, а поллер
сохранил его как `123…/Y` → не находит.

Для публичных групп (есть `@username`) оба пути дают `username/<msgid>` — там багов нет.
Поэтому проблема плавающая.

### Корневая причина B — неполная цепочка помечается завершённой

`context_pass._resolve_locally` ставит `thread_root_id = chain[-1].id` и
`context_fetched = True`, даже когда у верхнего локального узла (`chain[-1]`) ещё есть
`reply_to_tg_id` на родителя, которого нет в БД. Цепочка обрывается на последнем
локально известном узле, помечается завершённой, и сетевой догруз её уже не дочинит →
`thread_root_id` указывает не на настоящий корень.

## Решение

### 1. Единый canonical-namespace helper

Новая чистая функция в `backend/radar/core/providers/telegram.py` (рядом с парсерами):

```python
def chat_namespace(username: str | None, chat_id) -> str:
    """Канонический namespace чата для составного post_id.
    Один источник правды для поллера и realtime — одно сообщение даёт один post_id.
    Есть @username → username (без @, lower). Иначе → unmarked peer_id."""
```

- Unmarked peer_id получаем через `telethon.utils.resolve_id(int(chat_id))` — снимает
  префикс `-100` для супергрупп/каналов; для basic-групп id и так unmarked, функция
  возвращает его как есть.
- `chat_id` может прийти строкой (`#<id>`, `str(event.chat_id)`) — helper нормализует
  ввод (убирает `#`, парсит в int; на нечисловом вводе без username возвращает строку
  как есть, чтобы не падать).

Оба пути переводятся на helper:
- **Поллер:** `search_chat` уже резолвит entity; namespace считается из
  `entity.username` / `entity.id` через `chat_namespace`, результат передаётся в
  `_parse_tg_chat_message`. (Текущая передача `h` заменяется на canonical namespace.)
- **Realtime:** в `_on_message` `ns = chat_namespace(username, event.chat_id)` вместо
  `username or str(event.chat_id)`.

Итог: приватная группа `#1234567890` (поллер) и `event.chat_id = -1001234567890`
(realtime) → оба → `1234567890/<msgid>` → дедуп срабатывает, родители находятся.

### 2. Фикс RC-B (неполная цепочка)

В `_resolve_locally`:
- `thread_root_id` ставится **только** когда верхний достигнутый узел реально корневой
  (`node.reply_to_tg_id is None`).
- Если цепочка оборвалась из-за отсутствия родителя в БД (последний узел ещё ссылается
  на родителя) — **не** ставить `context_fetched = True`; вернуть значение, при котором
  `enrich_context` пойдёт в сеть и дочинит хвост на этом же тике.
- `reply_to_id` (прямой родитель) ставится как и раньше, когда непосредственный родитель
  локален — это корректно независимо от полноты остального хвоста.

### 3. Одноразовый скрипт схлопывания дублей

`backend/scripts/dedup_chat_namespace.py`:
- Найти упоминания, чей namespace — marked id (`-100…/…`), и сопоставить с близнецом,
  чей namespace — `resolve_id(marked)/<тот же msgid>`.
- Из пары оставить запись с непустым thread-контекстом (иначе — с меньшим id).
- Перевесить на оставшуюся: `IntelThreadContext.mention_id`, и у других упоминаний
  `reply_to_id` / `thread_root_id`, ссылавшиеся на удаляемую.
- Удалить дубль.
- `--dry-run` по умолчанию (печатает план); реальные изменения по явному флагу
  `--apply`. Бэкенд перезапускает ассистент (`launchctl kickstart -k
  gui/$(id -u)/com.echo.backend`), не пользователь.

### 4. Бэкфилл оборванных цепочек

В том же скрипте, отдельный режим/флаг: для упоминаний с `reply_to_tg_id IS NOT NULL`
и (`reply_to_id IS NULL` ИЛИ `thread_root_id` указывает на оборванный узел) сбросить
`context_fetched = False`, чтобы следующий тик `enrich_context` пере-собрал ветку уже с
правильным namespace. Батч ограничен (напр. 200 за прогон), чтобы не спровоцировать
flood-wait при последующем сетевом догрузе.

## Тесты (`python3 -m pytest` из `backend/`)

- **`chat_namespace`** (unit): `@username` → `username`; username-less marked
  `-1001234567890` → `1234567890`; basic-group unmarked id → без изменений; ввод `#<id>`
  и `str` нормализуются.
- **Регресс на дубль:** для одного сообщения username-less группы realtime-путь и
  поллер-путь дают одинаковый `post_id`.
- **`_resolve_locally`:** неполная цепочка НЕ помечается `context_fetched=True` и без
  `thread_root_id`; полная цепочка помечается завершённой с верным `thread_root_id`.
- **Дедуп-скрипт:** пара marked/unmarked схлопывается в одну строку, `IntelThreadContext`
  и ссылки `reply_to_id`/`thread_root_id` перевешиваются на оставшуюся.

## За рамками (YAGNI)

- LLM-классификация смысла реплая (подтверждение/опровержение/уточнение) — отдельная боль,
  не выбрана в этот заход.
- Изменение схемы БД / хранение стабильного идентификатора на пробе (вариант C) — не нужно
  для текущего фикса.
- Сиблинги в `_resolve_locally` — как и раньше, остаются на сетевом пути.
