# Tier-фильтр ключевых слов intel — дизайн

**Дата:** 2026-06-28
**Статус:** на ревью

## Проблема

Фильтр входа `keyword_relevant()` (`backend/radar/intel/collector.py`) впускает
пост в ленту по **одному** совпадению из словаря `intel_lexicon` (424 термина).
Памятка `_meta.usage_tips` в `keywords.seed.json` обещает «требуйте 2+ совпадения»,
но это так и не было реализовано — порог фактически = 1.

Замер по 32 558 сохранённым постам:

| #совпадений | постов |
|---|---|
| 0 | 2 403 (гео / аббревиатуры / куратор-слова) |
| **1** | **12 753 (39%)** |
| 2 | 12 890 |
| 3+ | 4 512 |

39% ленты впущено одним словом, и топ этих одиночек — многозначные generic-слова,
тянущие чистый мусор (доля sole-admit от всех вхождений термина):

- `сейчас` — 1522 одиночных впуска (77% вхождений) → политика, новости, шоу
- `очередь` — 325 (81%) → анекдоты, переговоры
- `работа`/`работают` — 474 (45–58%) → МКС, сбой Cloudflare
- `движение` — 259 (47%) → пробки на Крымском мосту
- `внимание`/`срочно` — 473 → реклама, налоги
- `громко`/`слышно`/`горит` — погода, бытовуха

При этом узкие военные термины поодиночке корректны: `бпла`, `шахед`, `дрон`,
`пво`, `вибух`, `прилет` — это легитимные короткие сводки, их терять нельзя.

## Цель

Срезать основную долю мусора из 12 753 постов-одиночек, **не теряя** короткие
легитимные сводки от сильных терминов. Решение — ввести у терминов **вес (tier)**.

## Правило впуска

Пост впускается, если выполнено **любое** из:

1. есть **≥1 STRONG**-совпадение, ИЛИ
2. есть **≥2 WEAK**-совпадения, ИЛИ
3. есть **1 WEAK + гео-метка** (`detect_direction`/`detect_place` нашли локацию).

Один WEAK без гео и без второго совпадения → **отбой**.

Существующий погодный guard (`AMBIGUOUS_WEATHER_TERMS` + `_looks_like_weather`)
сохраняется и применяется до подсчёта: погодные слова (`град`/`смерч`/`торнадо`)
в погодном контексте не идут в зачёт.

Куратор-управляемые positive-keywords (`spam_filter.load_keywords`,
`kind="keyword"`) **не меняются** — остаются мгновенным впуском (явная воля
куратора = strong), bypass length-gate как сейчас.

Аббревиатуры из `collector.ABBREVIATIONS` (`БПЛА`, `КР`, `УАБ`, `БЭК`, …) считаются
**STRONG** — они узкоспецифичны и матчатся только заглавными.

## Архитектура хранения

### 1. Колонка `tier` в `intel_lexicon`

- Модель `IntelLexicon` (`backend/radar/intel/models.py`): добавить
  `tier: Mapped[str] = mapped_column(Text, nullable=False, default="weak")`.
- Миграция: в `_MIGRATIONS` (`backend/radar/core/db.py`) добавить запись
  `"intel_lexicon": {"tier": "TEXT NOT NULL DEFAULT 'weak'"}`.
  Механизм идемпотентен, ALTER выполняется на старте только если колонки нет.
- Значения: `'strong'` | `'weak'`. Дефолт = `'weak'` (консервативно: новый
  непроверенный термин в одиночку ленту не зальёт).

### 2. Разметка в `keywords.seed.json` (источник истины)

Гибрид «категория + исключения»:

- у каждой категории — поле `"tier": "strong"|"weak"` (дефолт для всех её слов);
- в `_meta` — два списка точечных переопределений:
  - `"overrides_weak"`: слова из strong-категории, которые на деле слабые;
  - `"overrides_strong"`: слова из weak-категории, которые на деле сильные.

Итоговый tier термина = override (если есть) иначе tier категории.

### 3. Загрузка tier — `intake.ingest_lexicon_json`

При апсерте (`backend/radar/intel/intake.py`) вычислять итоговый tier
(категория → override) и писать в `intel_lexicon.tier`, в т.ч. обновлять tier у
уже существующих 424 строк (upsert уже идемпотентен — добавляем установку tier на
обеих ветках insert/update).

## Логика фильтра — `collector.py`

`matched_terms()` возвращает совпадения c tier. Карта `{term_lower: tier}`
грузится один раз на `collect_probe` рядом с текущим `lexicon_terms` (один запрос
`select term, tier from intel_lexicon`); аббревиатуры → tier `'strong'`.

`keyword_relevant(text, lexicon_tiers, geo_hit)` переписывается:

```python
hits = matched_terms(text, lexicon_tiers)         # [(term, tier)]
if not hits:
    return False
non_weather = [t for (t, _tier) in hits if t not in AMBIGUOUS_WEATHER_TERMS]
if not non_weather and _looks_like_weather(text):
    return False
strong = [t for (t, tier) in hits if tier == "strong"]
weak   = [t for (t, tier) in hits if tier == "weak"]
if strong:
    return True
if len(weak) >= 2:
    return True
if len(weak) == 1 and geo_hit:
    return True
return False
```

`geo_hit` (bool) прокидывается из уже вызываемого в `collect_probe`
`detect_direction`/`detect_place` — без дополнительных запросов. Сигнатура
`chat_message_relevant` тоже получает `geo_hit`; куратор-keyword ветка
(`kw_hit`) не меняется и проверяется раньше tier-логики.

## Стартовая разметка категорий (черновик по данным)

Обоснован частотой sole-admit. Пользователь правит при ревью.

| категория | default tier | обоснование |
|---|---|---|
| `missiles_weapons` | strong | узкие названия вооружений |
| `drones_uav` | strong | типы БПЛА |
| `air_defense_intercept` | strong | пво/сбили/перехват |
| `explosions_sounds` | strong | вибух/взрыв/прилёт — ядро |
| `consequences` | strong | специфичные фразы поражения |
| `ukrainian` | strong | укр. военный лексикон |

| `visual_effects` | **weak** | дым/огонь/горит/пожар бытовые |
| `movement_direction` | **weak** | летит/идёт/гул/свист generic |

  `свист`, `летить`, `летять`, `пролітає`, `небезпека`, `повітряна`, `повітряні`.

**`overrides_strong`** (слабая категория, но слово однозначно военное):

## Тестирование

`backend/tests/test_intel_keyword_tier.py` (новый):

- `keyword_relevant`: 1 strong → True; 1 weak без гео → False; 2 weak → True;
  1 weak + geo_hit → True; 0 hits → False; погодный guard (`град` в прогнозе) → False;
  `град` + взрыв → True.
- `intake`: tier из категории проставляется; `overrides_weak` делает слово weak в
  strong-категории; `overrides_strong` — наоборот; апсерт обновляет tier у
  существующей строки.
- `chat_message_relevant`: куратор-keyword по-прежнему впускает короткий ответ
  (bypass), tier-логика не ломает этот путь.

**Регресс-замер** (не тест, разовый скрипт): прогон новой логики по 32 558 постам —
сколько отсеялось/осталось. Ожидаемо: значимое сокращение 12 753 одиночек при
сохранении strong-сводок. Результат — в отчёт, не блокирует.

## Границы (явно вне scope)

- **Только вперёд:** существующие 32 558 постов не трогаем, ретро-чистки нет.
- Куратор-keywords (`kind="keyword"`) не меняются.
- UI/курирование tier из интерфейса — не делаем (правка через seed-файл).
