# Точечная заглушка сигналов — Design

**Дата:** 2026-06-28
**Ветка:** feat/source-subject

## Проблема

Сигналы (`IntelAlert`) генерируются чисто статистически: аномалия по объёму и/или
притоку источников у сюжета или направления. «Дежурные» официальные сюжеты
(поздравления, протокольные новости) регулярно дают высокий приток источников и
занимают ленту сигналов, хотя ценности не несут. Глобальная подкрутка порогов
(`ANOMALY_*` env) рубит чувствительность для всего сразу и всё равно не отсекает
такие темы (они часто превышают порог в 3+ раза). Нужна **точечная** заглушка
конкретного сюжета или целого направления, не трогая источники и не снижая общую
чувствительность.

## Решения (утверждены)

- **Охват:** глушим и отдельный сюжет, и целое направление.
- **Где кнопка:** в шторке сигналов (колокольчик) И в карточке сюжета.
- **Существующие сигналы:** при заглушке удаляются сразу из колокольчика; будущие
  блокируются. Есть UI для снятия заглушки («вернуть»).

## Архитектура

### 1. Модель данных
По аналогии с `IntelMention.hidden`:
- `IntelStory.muted` — `BOOLEAN NOT NULL DEFAULT 0`
- `IntelDirection.muted` — `BOOLEAN NOT NULL DEFAULT 0`

Оба регистрируются в `_MIGRATIONS` (`backend/radar/core/db.py`) — идемпотентный
`ALTER TABLE ADD COLUMN` при старте.

### 2. Логика скана (`backend/radar/intel/alerts.py`)
- `scan_story_alerts`: фильтр `IntelStory.muted == False`, плюс пропуск сюжетов,
  чьё направление заглушено (`direction.muted == True`).
- `scan_direction_alerts`: пропуск `d.muted == True`.

Заглушённое не доходит до `_emit`. Кулдаун (`ALERT_COOLDOWN_H`) не меняется.

### 3. Эндпоинты (`backend/radar/intel/api.py`)
- `POST /intel/stories/{id}/mute` → `story.muted=True` + удалить все `IntelAlert`
  с этим `story_id` + commit.
- `POST /intel/stories/{id}/unmute` → `story.muted=False`.
- `POST /intel/directions/{id}/mute` → `direction.muted=True` + удалить все
  `IntelAlert` с этим `direction_id` + commit.
- `POST /intel/directions/{id}/unmute` → `direction.muted=False`.
- `GET /intel/muted` → `{ "stories": [{id,title}], "directions": [{id,name}] }`.
- `aggregate.alert_payload` дополняется полем `direction_id` (сейчас отдаётся
  только `direction` key) — чтобы из сигнала знать, какое направление глушить.

Удаление (а не ack) выбрано намеренно: ack лишь помечает «прочитано», а цель —
убрать мусор из ленты.

### 4. Фронтенд (`echo-app/src/features/intel/`)
- **api.js**: `muteStory(id)`, `unmuteStory(id)`, `muteDirection(id)`,
  `unmuteDirection(id)`, `mutedList()`.
- **components/AlertBell.jsx**: у каждого сигнала кнопка 🔕; по `scope`
  (`story`/`direction`) зовёт нужный mute, сигнал убирается из списка оптимистично.
  Внизу шторки — сворачиваемая секция «Заглушено (N)» со списком сюжетов и
  направлений, у каждого кнопка «вернуть» (unmute).
- **components/IntelStories.jsx**: кнопка 🔕 в карточке/детали сюжета рядом с
  ✕-удалением; глушит сюжет, источник остаётся.

### 5. Тесты (`backend/tests/`)
- `scan_story_alerts` пропускает заглушённый сюжет.
- `scan_story_alerts` пропускает сюжет под заглушённым направлением.
- `scan_direction_alerts` пропускает заглушённое направление.
- mute-эндпоинт сюжета: ставит флаг + удаляет существующие `IntelAlert` этого сюжета.
- mute-эндпоинт направления: ставит флаг + удаляет существующие `IntelAlert`.
- unmute снимает флаг.
- `GET /intel/muted` возвращает оба списка.

## Вне охвата (YAGNI)
- Временная заглушка (на N часов) — только постоянная с ручным снятием.
- Заглушка по автору/каналу — это отдельный спам-механизм, уже существует.
- Изменение глобальных порогов `ANOMALY_*`.
