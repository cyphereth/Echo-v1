# Извлечение населённого пункта из текста поста

**Дата:** 2026-06-28
**Статус:** approved (design)

## Проблема

📍 в ленте/сюжетах берётся только из `subject` источника (`IntelProbe.subject` →
`tag_geo` → `IntelMention.subject`). Это работает для локальных чатов с заданным
куратором городом, но у бот-агрегаторов тревог (`@airalertmonitor_bot`,
`@ukrainealarmsignal` и т.п.) `subject` пуст — один бот покрывает десятки городов.
Локация («Воронеж», «Шостка») лежит в тексте поста, но выбрасывается: `detect_direction`
возвращает только область, теряя, какой именно город совпал.

Следствие — два изъяна:
1. У постов от бот-агрегаторов нет 📍 в ленте.
2. Тревоги по разным городам одной области («Шостка» и «Воронеж») сливаются в один
   сюжет — кластеризация идёт по `direction_id` (область) + сходству текста, а тексты
   тревог почти идентичны.

## Цели

- Извлекать населённый пункт из текста поста детерминированно (без LLM), переиспользуя
  существующий газеттир `geo.py`.
- Показывать извлечённый город в 📍 ленты и чип-фильтрах.
- Учитывать город при кластеризации: посты по разным городам не сливаются в один сюжет.

## Решение (приоритет — текст)

Если текст назвал город — он идёт в `subject`. Если текст молчит — fallback на `subject`
источника. Область по-прежнему решает текст (сохраняем текущее правило `tag_geo`).

### Компонент 1 — `geo.py`: газеттир с каноничными именами

Значения газеттира становятся парами *стем → отображаемое имя*:
```python
"sumy": {"судж": "Суджа", "шостк": "Шостка", ...}   # было: ["судж", "шостк", ...]
```
Стема недостаточно для показа («судж» ≠ «Суджа»), поэтому каждому стему сопоставляется
человекочитаемое имя.

Новая функция:
```python
def detect_place(text) -> tuple[str | None, str | None]:
    """(oblast_key, city) — первое совпадение в порядке словаря, как сейчас.
    city = каноничное имя совпавшего стема; None если ничего не найдено."""
```
`detect_direction(text)` остаётся тонкой обёрткой: `return detect_place(text)[0]` —
обратная совместимость, существующие вызовы (collector, retag_*) не меняются.

### Компонент 2 — `tag_geo`: город из текста

```python
city_key, city = detect_place(text)
if city_key:
    dir_id = resolve_direction_id(session, city_key)
    same = src_dir and dir_id == src_dir
    # текст важнее: его город приоритетен; иначе subject источника, если та же область
    return dir_id, (city or (subject if same else None))
# текст молчит — fallback на область+город источника
return (src_dir or resolve_direction_id(session, None)), subject
```
Поведение для локальных чатов без географии в тексте не меняется (fallback на источник).
Репост о другой области по-прежнему не получает ложный город источника.

### Компонент 3 — кластеризация по городу

`cluster_owner` получает необязательный предикат:
```python
def cluster_owner(..., match_guard: Callable | None = None):
```
`match_guard(candidate, mention) -> bool` вызывается дополнительно к проверке
`owner_field` и временно́го окна в `_attach_to_incident` / `_attach_to_story`. Дефолт
`None` → brand/news домены не меняются (domain isolation сохранена).

Intel (`stories.py`) передаёт guard, требующий совпадения `subject`:
```python
lambda cand, m: (getattr(cand, "subject", None) or None) == ((m.subject) or None)
```
Пост без города (`subject is None`) образует свою «нейтральную» корзину и не липнет к
городам. Инцидент/сюжет получают `subject` от первого вошедшего упоминания (новое поле
`subject` на `IntelIncident`/`IntelStory`, nullable; нужна лёгкая миграция/ensure-column).

## Данные / схема

- `IntelIncident.subject` (nullable str) — новое.
- `IntelStory.subject` (nullable str) — новое.
- Заполняются в `_attach_to_incident` / `_attach_to_story` при создании из первого
  упоминания; guard сверяет с ним кандидатов.

## Тесты

- `detect_place`: область+город по стему; склонения («в Судже» → Суджа); нет совпадения → (None, None).
- `detect_direction`: обёртка по-прежнему отдаёт только ключ (регрессия).
- `tag_geo`: текст с городом → город из текста; текст молчит → город источника; репост о
  другой области → нет ложного города.
- кластеризация: два поста с одинаковым `subject` мёржатся; с разным — нет; `None` не липнет к городу.

## Не входит (YAGNI)

- LLM-NER по тексту (улицы, нестандартные топонимы вне газеттира).
- Множественные города в одном посте (берём первое совпадение, как сейчас область).
- Бэкфилл `subject`/`subject`-кластеров для уже сохранённых постов (можно отдельно через
  `retag_*`).

# Расширение: газетир на всю РФ/Украину + ключевые направления на доске

**Дата:** 2026-06-28 (extension)
**Статус:** approved

## Проблема

После внедрения 📍 из текста остались два пробела:

1. **Короткие укр-посты** («Загроза. Суми») не переводятся: порог `maybe_translate`
   (`≥2` укр-специфичных буквы `і/ї/є/ґ`) не срабатывает, текст остаётся украинским, а
   газетир хранил в основном русские формы → 📍 не ставится.
2. **Покрытие.** Газетир заточен под приграничье РФ + фронт. Города вглубь РФ (Москва,
   Казань…) и центр/запад Украины (Полтава, Львов, Одесса…) отсутствуют — у бот-агрегаторов
   тревог они мелькают постоянно, а 📍 пустая.

## Решение

### Компонент A — двухслойный газетир (`geo.py`)

- `GEO_KEYWORDS` (направление → {стем: город}) расширяется до **всех областей РФ и
  Украины, привязанных к ключевым фронтам**; у каждого города и **русская, и украинская**
  форма-стем (`"сумы"`/`"суми" → "Сумы"`, `"київ"/"киев" → "Киев"`). Детект перестаёт
  зависеть от перевода → чинит короткие укр-посты.
- Новый `PLACE_ONLY: dict[str, str]` (стем → город) — города **вне ключевых фронтов**
  (Москва, Львов, Одесса, Винница, Казань…). Дают 📍, но направление = `None`.

`detect_place` сперва сканирует `GEO_KEYWORDS` (даёт `(key, city)`), затем `PLACE_ONLY`
(даёт `(None, city)`). Порядок сохраняет приоритет фронтовых стемов над «прочими».

### Компонент B — `tag_geo`: 📍-only города

Добавляется ветка: `key is None`, но `city` найден →
`return (src_dir or unassigned), city` — 📍 показывается, направление падает в unassigned.

### Компонент C — оперативная доска (фронт, `IntelBoard.jsx`)

Доска показывает только **ключевые** направления, остальные (`dnipro`, `kyiv`,
`chernihiv`, `unassigned`) скрыты. Приграничные области РФ (`bryansk`, `belgorod`,
`kursk`, `voronezh`, `oryol`, `rostov`) остаются **отдельными** направлениями в БД, но на
доске **группируются** под заголовком «Приграничье РФ» (группировка только в UI). Бэкенд
`/intel/directions` не меняется (риск для других потребителей).

Порядок групп на доске: «Приграничье РФ» (6 областей), затем Сумское, Харьковское, ЛНР,
ДНР, Запорожское, Херсонское, Крым.

## Решения по вопросам

- Охват: **вся РФ + вся Украина** для 📍.
- Направления на доске: только ключевые; «Приграничье РФ» = группа отдельных областей (UI).
- Города вне ключевых: 📍 в ленте, направление = unassigned.
- Порог перевода **не трогаем** — двуязычный газетир делает детект независимым от перевода.

## Тесты

- `detect_place`: укр-форма без перевода («Суми» → ("sumy","Сумы")); `PLACE_ONLY`
  («ракета над Москвой» → (None,"Москва")); «погода» → (None,None) (регрессия).
- `tag_geo`: 📍-only город без направления → (unassigned, город); `subject` не теряется.
- Существующие geo/tag_geo тесты остаются зелёными.

## Не входит (YAGNI)

- Новые `IntelDirection` под Москву/Львов и т.п. (они в unassigned).
- Понижение порога перевода.
- Бэкфилл уже сохранённых постов.
