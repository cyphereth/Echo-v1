## 1. Что строим

B2B-инструмент репутационного радара для брендов в **Instagram и TikTok** (рынок РФ/СНГ).
Ловим публичные упоминания бренда, классифицируем по тону и виральности, и отдаём
менеджеру PR/SMM карточку с готовым черновиком ответа. Уникальность — не «ещё один
мониторинг», а **раннее предупреждение о виральности**: предсказываем, залетит ли
негативный ролик, пока он не набрал охват.

Сценарий-эталон: блогер жалуется на испорченный заказ, ролик начинает расти →
система ловит это на разгоне → пуш в PR за минуты с черновиком ответа → бренд тушит
до того, как это стало кризисом.

## 2. Принятые решения (НЕ пересматривать без явной просьбы)

- **Авторизация v1:** один аккаунт на бренд. Роли (PR/SMM/admin) — v2.
- **AI-онбординг зондов:** клиент может дать URL сайта/соцсетей → продукт парсит
  страницу и AI предлагает ключевые слова и хэштеги для зондов. Клиент правит и
  подтверждает. Если URL не дан — настраивает вручную через веб-интерфейс.
- **Confidence-гейт:** в v1 fallback-порог `< 0.6` → нет черновика, упоминание
  помечается «решает человек». Порог вытесняется tone-learning по мере накопления правок.
- **Tone-learning:** каждая правка черновика логируется в `draft_edits`
  (`mention_id`, `brand_id`, `category`, `original`, `edited`, `timestamp`).
  В v1 логируем только. В v2+ правки подставляются как few-shot примеры в промпт.
  При онбординге клиент может загрузить 3–5 примеров «идеального ответа» — AI
  использует их сразу, не ждёт накопления. Несколько менеджеров — правки копятся
  по бренду без разделения по людям.
- **Telegram-пуш:** ссылка на пост + тональность (негатив/позитив) + Severity score
  (виральность) + черновик ответа.
- **Лимиты упоминаний:** считаем по уникальным `mentions` (не снимкам). При
  достижении лимита уведомляем клиента. Дополнительно: считаем среднее потребление
  за последние 7 дней → если прогноз говорит «лимит через ≤ 1 день» → алерт клиенту.



- **Площадки v1:** только Instagram + TikTok. Архитектура коннекторов расширяема
  (VK/YouTube/Telegram/Яндекс.Карты — потом), но запуск строго на IG + TikTok.
- **Сбор — через арендованные data-API**, свой скрейпер/прокси НЕ строим:
  TikTok → EnsembleData (или TikHub/Apify на старте без месячного порога),
  Instagram → HikerAPI. Грязный слой (прокси, антибот) арендуем.
- **Решение всегда за человеком.** В v1 НЕТ автопубликации вообще. Продукт
  предлагает черновик — менеджер одобряет / правит / отклоняет.
- **Юмористические ответы НИКОГДА не автопостятся** (даже в будущем). Только ручное
  подтверждение SMM.
- **Confidence-гейт:** при низкой уверенности классификатора в тоне черновик НЕ
  предлагается, упоминание помечается «решает человек».
- **Покупатель — PR + SMM** (часто один отдел). Продажников НЕ трогаем в v1.
- **Tone-learning:** каждую ручную правку черновика логируем по бренду с первого дня
  (актив на будущее), но дообучение — не в v1.

## 3. Стек и архитектура

- Python 3.12, **FastAPI**, **SQLAlchemy 2.0**, **APScheduler** (v1; позже ARQ/Redis).
- БД: локально SQLite, на **Railway — Postgres** через `DATABASE_URL` (см. `radar/db.py`).
- LLM: каскад моделей — дешёвая на первичный триаж всего потока, дорогая только на
  пограничные и high-severity кейсы. Доступ через `httpx`.
- Доставка: Telegram-бот для срочных PR-алертов + веб-инбокс для остального.

Поток данных (конвейер):
`Сбор по расписанию → Дедуп + матч бренда → Классификация (LLM) → Скоринг →
Черновик (LLM) → Маршрут + доставка`, плюс цикл **hot-watch**: упоминания, перешедшие
порог роста, возвращаются на учащённый перезапрос для пересчёта скоринга.

## 5. Жёсткие инварианты (нарушать нельзя)

1. **Watermark двигается ТОЛЬКО после успешного ingest.** Запрос упал → ретрай с
   backoff, метка на месте. Потеря метки = дыра в данных или повторная оплата API.
2. **Дедуп через upsert по `UNIQUE(platform, post_id)`.** Повтор обновляет метрики,
   не плодит карточки (один пост приходит и по keyword-, и по hashtag-зонду).
3. **Тихий проход стоит 1 запрос.** Тянем первую страницу, видим, что новее watermark
   ничего нет — выходим. Глубокая пагинация только когда реально есть новое.
4. **Hot-watch только для `is_hot` упоминаний.** Никогда не перезапрашиваем весь поток.
5. **Severity пересчитывается на каждом тике hot-watch**, не только при ingest.
6. **acceleration важнее velocity** в скоринге (ловит до пика). При <3 снимках
   ускорение не выдумывается — возвращаем 0 и фазу `rising`/`unknown`.
7. **Юмор не автопостится. Низкая уверенность → нет черновика.**

## 6. Дисциплина затрат (экономика → код)

COGS живой (платим за запрос к API), поэтому в коде:
- **Адаптивный интервал зондов**, не фиксированный: всплеск/high-severity → чаще,
  тишина → реже. + вес по времени суток.
- **Jitter при запуске зондов** + собственный token-bucket на исходящие вызовы —
  чтобы не словить rate-limit провайдера (никакого thundering herd в начале часа).
- **Комментарии под виральными постами — сэмплировать** (первые N + случайная
  выборка), НИКОГДА не выкачивать все: один пост с 10k комментов разорит месяц.
- **Каскад LLM:** дешёвая модель на весь поток, дорогая — только пограничные/high-sev.
- Тариф клиента режет **лимит обрабатываемых упоминаний/мес** (не флэт).

## 7. Фазы сборки (по порядку; каждая — отдельная задача с критерием приёмки)

### P1 — Провайдеры (абстракция сбора)
- `radar/providers/base.py`: интерфейс `SearchProvider` с методом
  `search(query, kind, cursor) -> SearchPage(posts, next_cursor)`. Унифицированная
  модель поста (post_id, author, followers, text, hashtags, created_at, метрики, sound_id).
- `radar/providers/mock.py`: мок-провайдер с детерминированной выдачей для тестов без сети.
- **Приёмка:** тест, который листает мок-провайдер курсором и собирает N постов.

### P2 — Коллектор (широкий проход)
- `radar/collector.py`: цикл по due-зондам → `search` с пагинацией до watermark →
  upsert в `mentions` → первый `mention_snapshot` → сдвиг watermark ПОСЛЕ ingest.
- Матч бренда + отсев омонимов/шума перед записью.
- **Приёмка:** на мок-провайдере второй проход даёт 0 новых; дубль по двум зондам
  не плодит строки; watermark на месте при смоделированной ошибке.

### P3 — Шедулер
- `radar/scheduler.py`: APScheduler-тик, выбор due-зондов, адаптивный интервал, jitter,
  token-bucket.
- **Приёмка:** зонды запускаются по `next_run_at`; интервал меняется по активности.

### P4 — Hot-watch
- `radar/hotwatch.py`: учащённый перезапрос только `is_hot`, новый снимок,
  пересчёт Severity и фазы, снятие с hot-watch при фазе `declining`.
- **Приёмка:** растущий пост набирает снимки и обновляет Severity; затухший снимается.

### P5 — Классификация (LLM)
- `radar/classify.py`: тон + уверенность, категория (viral_negative/complaint/
  neutral/positive/casual), каскад моделей. Confidence-гейт.
- **Приёмка:** категория и lane проставляются; на низкой уверенности lane=none, без черновика.

### P6 — Черновики ответов (LLM)
- `radar/drafts.py`: генерация под тон категории, ОБЯЗАТЕЛЬНО с конкретным следующим
  шагом (не «нам жаль», а «пришлите номер заказа»). Юмор помечается «только вручную».
- **Приёмка:** черновик содержит next-step; для юмора стоит флаг ручной публикации.

### P7 — API + инбокс
- `radar/api.py`: FastAPI. Эндпоинты инбокса (две полосы PR/SMM), карточка упоминания,
  действия одобрить/править/отклонить, лог правок (tone-learning).
- Telegram-бот для срочных PR-пушей.
- **Приёмка:** инбокс отдаёт карточки по полосам; правка логируется; PR-пуш уходит.

## 8. Вне scope v1 (НЕ строить)

- Видео-контекст (ASR/OCR) — только текст: подписи, комменты, хэштеги.
- Режим продаж/исходящий lead-gen — другой фильтр и выход, отдельный режим v2.
- Любая автопубликация.
- VK/YouTube/Telegram/Яндекс.Карты как источники.
- Сравнение с конкурентами, мультибренд-дашборды, аналитика.

## 9. Тестирование и конвенции

- Тесты гоняются **без сети** через мок-провайдер. Не ходить в реальные API в тестах.
- Реальные ключи API — через env (`ENSEMBLE_TOKEN`, `HIKER_KEY`, `LLM_API_KEY`), никогда в коде.
- Прод-миграции — Alembic (после P2). `init_db()` из `db.py` — только для локалки.
- Все денежные/числовые величины наружу — округлять.
- Держать `scoring.py` без внешних зависимостей.
